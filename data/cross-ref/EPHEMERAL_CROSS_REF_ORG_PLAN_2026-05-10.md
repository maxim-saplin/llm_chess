# Ephemeral Cross-Ref Org Plan

Status: ephemeral planning note for the 2026-05-10 review.

Do not treat this file as part of the published cross-ref research surface.
Do not reference it from README, skills, result artifacts, or source packages.
Do not reuse the leaked `CROSS_REF_RESEARCH_PLAN.md` naming pattern.

## Intent

Reorganize `data/cross-ref/` so a cold reviewer can answer five questions quickly and correctly:

1. What is the run-time mapping source of truth?
2. Where is the best entry point to review `llm_chess` to benchmark mappings?
3. How do I rerun or update an existing eval without mutating published outputs by accident?
4. How do I add a new benchmark without reverse-engineering the framework?
5. Which outputs, commentary, and review steps are generated by agents from data, which remain manual, and how are those boundaries plus override points enforced?

## Key Goals

1. Put mapping first in the navigation and make the source of truth unmistakable.
2. Add a reviewer-friendly, model-centric mapping surface instead of forcing users to inspect raw CSVs and per-eval HTML separately.
3. Make mapping artifacts sane to review and edit by keeping the source model identifier and destination `llm_chess` identifier adjacent and near the front of mapping files.
4. Make repeatability operational with executable runbooks, clear CLI help, a dry-run path, and explicit user-invokable pipeline steps.
5. Add an agentic analyst playbook for reruns, diffs, report generation, and benchmark onboarding, with clear human review and manual override points.
6. Make narrative generation auditable so agent-generated commentary is derived from published summary data, diff artifacts, and declared inputs rather than silent hand editing.
7. Preserve user control over key stages: running scripts directly, reviewing intermediate artifacts, fixing mappings manually, and rerunning only the affected stages.
8. Reduce the marginal cost of adding a new benchmark by documenting and templating the required contracts.

## Top Priorities

### Priority 1: Fix the doc taxonomy and start-here surface

Problem:
The current workspace has real machinery, but the first-reader path is still scattered across `README.md`, `CONSOLIDATED_REPORT.md`, mapping notes, result HTML, and unstated implementation details.

Deliverables:

1. Rewrite the README around user tasks, not folder inventory.
2. Start the README with `mapping source of truth`, `review here`, `rerun here`, and `add-a-benchmark here`.
3. Remove stale references to leaked plan files from durable docs and skills.
4. Introduce a small, durable maintenance doc set with explicit purposes.

Target durable docs:

1. `README.md`: entry point and navigation.
2. `REVIEW_GUIDE.md`: how to inspect mappings and published results.
3. `OPERATIONS.md`: dry-run, publish, and update-existing-eval workflows.
4. `ADD_BENCHMARK.md`: benchmark onboarding contract and checklist.
5. `AGENT_PLAYBOOK.md`: supported agentic asks, stage boundaries, and human override points.

Success criteria:

1. A reviewer can find the mapping source of truth in under 30 seconds.
2. No durable doc points to a missing planning artifact.
3. The README no longer depends on private context or prior authorship knowledge.

### Priority 2: Add a single mapping review entry point

Problem:
The mapping exists, but the review surface is eval-centric and fragmentary. There is no convenient way to inspect one `llm_chess` player across benchmarks or review unresolved mappings from one place.

Deliverables:

1. Add a model-centric mapping review artifact generated from the mapping CSVs.
2. Add a CLI entry point that can emit the artifact and filter it.
3. Support both eval-centric and model-centric review flows.
4. Reorder the mapping contract so the source-side model label/id and destination `llm_chess_player` sit adjacent near the front, with rationale and review metadata after the core mapping fields.

Target outputs:

1. A joined mapping matrix CSV keyed by `llm_chess_player` and `eval_id`.
2. An HTML review surface for unresolved, ambiguous, unmatched, and excluded rows.
3. Optional filtered views by player, eval, status, or provider.
4. Mapping CSVs whose first editing columns make the mapping itself obvious before any review metadata.

Recommended CLI shape:

1. `run_cross_ref.py mapping-review`
2. Optional flags: `--eval-id`, `--player`, `--status`, `--provider`, `--html-output`, `--csv-output`

Success criteria:

1. A reviewer can inspect one model across all current benchmarks without opening multiple files manually.
2. A reviewer can inspect unresolved rows without reading raw CSVs first.
3. The artifact is generated from the run-time mapping CSVs, not from a separate shadow source.
4. A human can manually fix a bad mapping without horizontally scanning through low-value metadata columns first.

### Priority 3: Make repeatability and agentic operations explicit and low-risk

Problem:
The current runner works, but the operator workflow is still implicit. Help text is thin, dry-run guidance is easy to miss, diffing changes after score updates is not first-class, and the add-a-benchmark flow is not encoded as an agent-plus-human playbook.

Deliverables:

1. Expand CLI help text for every eval subcommand and new review commands.
2. Document a standard non-mutating dry-run flow.
3. Document a publish flow with required verification evidence.
4. Add an explicit agentic playbook for the two primary asks:
	- "I added new scores; rerun the cross-eval, regenerate reports, and tell me what changed."
	- "I added a new dataset; integrate it, map it, run the analysis, and update the consolidated picture."
5. Define user checkpoints for running scripts manually, reviewing intermediate artifacts, editing mappings, and rerunning only the affected stages.
6. Add a narrow smoke test for documented commands where practical.

Required workflows:

1. Refresh inventory only.
2. Dry-run one eval to temporary outputs.
3. Publish one eval to `results/`.
4. Refresh all currently supported evals.
5. Rerun after changed scores and emit machine-readable plus narrative diff artifacts.
6. Add a new benchmark with agent-proposed artifacts, explicit review checkpoints, and final publication steps.
7. Verify published artifacts after regeneration.

Success criteria:

1. A maintainer can rerun ECI or ARC without reading adapter code.
2. The supported agentic asks are documented as concrete workflows rather than implied behavior.
3. A user can stop after mapping, review or edit artifacts manually, and continue without restarting the whole pipeline.
4. Dry-run and publish commands are visibly different in the docs.
5. The documented commands are executable against the current repo state.

## Remaining Plan

### Workstream 4: Operationalize agentic analyst workflows and narrative generation

Problem:
Per-eval summaries are data-backed, but the agentic workflow around reruns, diffs, commentary, and benchmark onboarding is still implicit. Prose is only the most visible symptom. The bigger issue is that the system does not yet define which stages are automated, which are agent-drafted, where human review belongs, and how a user can intervene without losing the workflow.

Plan:

1. Define a stage graph for the core agentic workflows: rerun existing evals, rerun after score updates, and add a new benchmark.
2. Define which stages are script-driven, which stages are agent-driven, and which stages require or permit human review.
3. Generate a consolidated machine-readable cross-eval summary from published per-eval summaries.
4. Generate a first-pass consolidated report and a first-pass diff report from declared summary inputs.
5. Generate machine-readable diff artifacts whenever scores, mappings, or coverage change.
6. Keep any manual interpretation in a clearly delimited section with explicit editing rules and provenance.

Primary agentic asks this playbook must support:

1. "I added new scores; rerun the cross-eval, regenerate the reports, and tell me what changed."
2. "I added a new dataset; integrate it, propose mappings, run the analysis, and update the consolidated picture."
3. "Pause after mapping generation so I can fix questionable rows manually, then continue."

Recommended output split:

1. `results/*_summary.json`: authoritative per-eval machine-readable facts.
2. `results/cross_ref_summary.json`: generated cross-eval aggregate facts.
3. `results/*_diff.json` and `results/*_diff.html`: generated change surfaces for reruns and score updates.
4. `results/cross_ref_report.md` or `.html`: generated narrative and tables.
5. Optional curated commentary layer only if it adds meaning not already present in the generated report.
6. `AGENT_PLAYBOOK.md`: user-facing workflow contract describing supported asks, checkpoints, and override paths.

Success criteria:

1. Every headline claim in generated commentary is traceable to a published summary field or diff artifact.
2. A rerun updates the consolidated report and the change report mechanically.
3. The system can explain what changed after score updates, not just regenerate static reports.
4. Manual prose, if retained, is explicitly labeled as manual interpretation.
5. A reviewer can tell whether an artifact is script-generated, agent-generated, manually edited, or manually curated.

### Workstream 5: Add a benchmark onboarding path

Problem:
Adding a new benchmark currently requires understanding the adapter shape, source normalization contract, mapping contract, output contract, and validation path from code.

Plan:

1. Document the minimum required files and contracts for a new eval.
2. Provide an adapter checklist, a code template, and an agentic playbook for the add-a-benchmark flow.
3. Provide an end-to-end onboarding checklist from source snapshot to published outputs.
4. Define review checkpoints for source-contract validation, mapping proposal review, post-run QA, and consolidated-output update.
5. Add tests that fail clearly when a new adapter breaks the shared contract.

Required onboarding checklist:

1. Add source snapshot and `SOURCE.md`.
2. Implement normalization.
3. Add or seed mapping CSV.
4. Register adapter in the runner.
5. Review or manually edit proposed mappings before publication when needed.
6. Generate dry-run outputs.
7. Run focused tests.
8. Publish final outputs with verification metadata.
9. Regenerate the cross-eval review and diff surfaces.
10. Update the consolidated agent-generated outputs.

Success criteria:

1. A new benchmark can be added without copying hidden conventions from existing adapters.
2. The required contract is visible before any code is written.
3. The add-a-benchmark workflow exposes where the user can inspect artifacts or intervene manually.
4. Missing steps fail in tests or docs, not in reviewer memory.

### Workstream 6: Enforce artifact and control-surface boundaries

Problem:
Research outputs, maintenance docs, agent-generated commentary, manual edits, and planning artifacts can blur together. That makes it hard to tell what is authoritative, what is generated, what is editable by a user, and what is only temporary process support.

Plan:

1. Define durable categories and their allowed contents.
2. Keep `results/` as the published output surface only.
3. Keep `evals/` for source snapshots and provenance notes only.
4. Keep maintenance docs and the agent playbook in a dedicated durable doc surface.
5. Keep manual-review surfaces explicit so mapping fixes and curated commentary are visible interventions rather than silent edits.
6. Keep ephemeral planning files explicitly named and unreferenced.

Boundary rules:

1. No ephemeral file is linked from README, skills, or results.
2. No result artifact points to temporary planning docs.
3. Generated report artifacts must declare their source summaries, diff inputs, and generation path.
4. Manual overrides must happen in explicit review surfaces such as mapping files or curated notes, not as silent edits to generated outputs.
5. No source package contains implementation or planning material.
6. Durable docs may reference runbooks and output surfaces, but not temporary decision notes.

Success criteria:

1. The workspace tells a consistent story about what is published, what is generated, what is manually curated, what is operational, and what is temporary.
2. A reviewer can tell where human intervention is allowed and how it is recorded.
3. Future planning notes cannot accidentally become part of the research surface by naming convention alone.

## Acceptance Criteria For The Reorg

1. The first screen of the README answers `what is the mapping source of truth` and `where do I start reviewing`.
2. There is a single generated mapping review surface that works across all supported evals.
3. Mapping files keep the source-side model identifier and destination `llm_chess` identifier adjacent near the front.
4. A maintainer can execute dry-run, publish, rerun-plus-diff, and score-update workflows from the docs without reading adapter code.
5. The system supports documented agentic asks for `rerun and explain the diff` and `add a new benchmark and update the picture`, with clear user checkpoints.
6. The consolidated report and its commentary are generated from published summaries and diff artifacts or clearly labeled as manual.
7. A new benchmark path exists as a checklist, template, and playbook rather than tribal knowledge.
8. No durable doc references missing or temporary planning files.

## Non-Goals

1. Reworking the statistical methodology of existing evals.
2. Expanding the supported benchmark set before the maintenance surface is fixed.
3. Folding temporary planning artifacts into the canonical cross-ref documentation.
